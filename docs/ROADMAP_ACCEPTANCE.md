# Roadmap Acceptance Criteria

目的：为 `TODO_ROADMAP.md` 中各项能力提供统一、可执行、可回归的验收口径，避免“功能看起来做了但无法稳定证明”。

## 1) 验收总原则

- **先语义正确，再性能优化**：任何性能结论都必须建立在正确性测试通过的前提上。
- **先可复现，再可扩展**：所有验收都需给出命令、输入参数、预期输出指标。
- **指标要有上下文**：延迟、吞吐、丢包等指标必须记录测试场景（并发、消息大小、注入条件、CPU 绑定策略）。
- **至少双维度验收**：每个里程碑至少包含“正确性 + 可观测性”两类证据。

## 2) 功能清单映射

按模块分三层：

- **基础设施层**
  - 队列/ring/内存池契约：容量、生命周期、并发模型、边界行为。
  - IO 可中断语义：超时、取消点、错误码一致性。
- **协议语义层**
  - UDP：解析/序列化、校验和、长度与畸形包处理。
  - TCP：状态机、seq/ack、重传、流控、关闭流程。
- **系统集成层**
  - netns 场景回归（丢包/乱序/延迟注入）。
  - 多后端一致性（epoll/io_uring/busy poll）。

## 3) 指标口径（默认）

- **正确性指标**
  - 报文收发成功率：`success / total`
  - 协议错误率：`protocol_error / total`
  - 序号连续性：`gaps`、`reorder_events`、`max_reorder_depth`
- **时延指标**
  - 端到端延迟：至少输出 `p50/p99/p999/max`
  - 测量窗口：显式记录样本数与持续时间
- **吞吐指标**
  - 每秒消息数（msg/s）或字节速率（MB/s）
  - 在固定 CPU/线程绑定下重复至少 3 次，记录中位值
- **稳定性指标**
  - 长跑时长（建议 >= 5 分钟）
  - 异常计数：超时、重连、断连、背压命中

## 4) 测试准入规则（Gate）

对每个 TODO 子项，至少满足以下 Gate：

1. **单元测试 Gate**
   - 关键路径必须有单元测试覆盖（含边界条件）。
   - 新增测试在本地可直接运行，且可重复通过。
2. **场景测试 Gate**
   - 至少 1 个集成或仿真场景覆盖该能力。
   - 输出与该能力强相关的指标（例如重传项必须输出 timeout/retry 统计）。
3. **文档 Gate**
   - 在 docs 或 roadmap 中明确“做了什么、如何验证、已知限制”。
4. **回归 Gate**
   - 不破坏已有核心测试（`ctest` 关键集合通过）。
