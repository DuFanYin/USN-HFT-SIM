# run_sim_market — 行情仿真

`run_sim_market` 模拟一个最小化的 **UDP 组播行情链路**：一个发布端持续向组播地址广播行情快照，一个订阅端实时接收并检测序号连续性。

## 仿真场景

| 角色 | 做什么 |
| --- | --- |
| 发布端 (feed_publisher) | 每 500 ms 向组播组 `239.0.0.1:9100` 发送一条行情增量消息，包含合约 10001 的买卖价量与递增序号 |
| 订阅端 (feed_subscriber) | 加入同一组播组，逐条接收行情，跟踪序号是否连续；若发现跳号则输出 GAP 警告，每 10 条打印一次统计摘要 |

整个仿真就是一条 **"发布 → 组播 → 接收"** 的单向数据流，用来验证 UDP 行情通道是否工作正常。

## 行情消息内容

每条行情包含以下字段（固定值，不做随机化）：

| 字段 | 值 | 含义 |
| --- | --- | --- |
| seq | 从 1 递增 | 全局序号，供订阅端做连续性校验 |
| instrument_id | 10001 | 模拟合约 ID |
| bid_price | 100000 | 买一价（1e-4 精度，即 10.0000） |
| bid_qty | 10 | 买一量 |
| ask_price | 100010 | 卖一价（10.0010） |
| ask_qty | 10 | 卖一量 |

## 运行方式

```bash
# 默认运行 10 秒
./build/apps/run_sim_market

# 指定运行 30 秒
./build/apps/run_sim_market 30
```

## 生命周期

1. **启动订阅端** — 先让订阅端就绪，确保不会错过最早的行情
2. **等待 300 ms** — 给订阅端留出加入组播组的时间
3. **启动发布端** — 开始广播行情
4. **运行 N 秒** — 发布端持续发送，订阅端持续接收和统计
5. **停止** — 先停发布端，再停订阅端，全部回收后退出

## 观察要点

- **正常情况**：订阅端持续收到行情，序号无跳号，每 10 条输出一行统计
- **GAP 告警**：如果网络丢包或订阅端启动晚于发布端，会出现序号跳跃的 GAP 日志
- **发送频率**：约 2 条/秒（500 ms 间隔），10 秒运行约产生 20 条行情
